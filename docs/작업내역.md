## 2026-07-15 — 수집 메일 → 에이전트(Hermes) 업무 분해 (Gemini와 병렬)

> **상태:** 미커밋. `/mails` "업무로 등록" 모달에 기존 **Gemini** 분석 옆에 **자체호스팅 Hermes 에이전트** 분석을 나란히 추가.
> **검증:** `npm run build`(타입체크) 통과, 변경 파일 `eslint` clean(전체 lint 실패는 기존 graph/kanban 부채와 무관), code-review 반영. Discord 경유 **재실측 왕복 ≈21.5초**로 JSON 스키마 준수 확인, 브라우저 E2E로 에이전트 결과가 우측 컬럼에 표시되는 것까지 실측.

수집 메일을 업무로 분해할 때 **Gemini**(즉시)와 **Hermes 에이전트**(Hostinger VPS, Codex 백엔드)의 결과를 **좌/우 2컬럼**으로 비교·선택. Hermes는 **Discord REST 경유**만 가능하고 왕복이 길어(수십초~수분) 동기 호출이 불가하므로, **전송→폴링 2단계 비동기**(화면에서 대기하며 자동 갱신)로 구현. 등록은 기존 `registerMailTasks`를 재사용(양쪽에서 고른 업무를 합쳐 넘기면 제목 dedup 적용).

| 파일 | 역할 | 핵심 |

| `src/server/agent/discord.ts` (신규, `server-only`) | Discord REST v10 유틸 | `postAgentMessage`·`fetchMessagesAfter`(429 재시도)·`extractJson`·`cmpId`(BigInt 리터럴 없이 길이+사전순)·`agentChannelId`·`isAgentAvailable`. 봇 토큰은 서버에서만 |

### 근본 이슈와 해결 (인프라 — 코드 밖)
- **문제:** 앱이 Hermes와 **같은 봇 토큰**을 쓰면, Hermes의 `on_message`가 `if message.author == self._client.user: return`으로 **자기 메시지를 무조건 무시** → 앱 전송이 트리거 안 됨(초기 실측 6분 무응답의 원인).
- **해결:**
  1. 앱 전용 **별도 Discord 봇**(`PMS Agent Bridge`) 생성 → Hermes가 있는 서버에 초대(채널 보기·메시지 보내기·메시지 기록 읽기) → 로컬 `.env`의 `DISCORD_BOT_TOKEN`을 새 봇 토큰으로 교체.
  2. Hermes VPS `.env`에 **`DISCORD_ALLOW_BOTS=all`** 추가 후 `docker compose restart`. 봇 메시지를 human allowlist에서 우회 허용(`gateway/authz_mixin.py`의 `DISCORD_ALLOW_BOTS ∈ {mentions, all}` 경로). VPS 경로: hPanel 웹터미널, `.env`=`/docker/hermes-agent-wtnq/data/.env`, 코드=컨테이너 `/opt/hermes`.

### 주의 / 후속 과제
- **화면 대기 방식이라 모달을 닫으면 결과 유실.** 견고화하려면 요청을 DB에 저장 + cron 수거(google-sync cron 패턴 재사용).
- **배포(Vercel):** `DISCORD_BOT_TOKEN`·`DISCORD_AGENT_CHANNEL_ID` 환경변수 반영 필요.
- `DISCORD_ALLOW_BOTS=all`은 해당 채널의 **다른 봇 메시지에도** Hermes가 반응할 수 있음(전용 채널 권장).
- 이전 실측(2026-07-15 오전) 대비 왕복이 272초→21.5초로 크게 단축(순수 JSON 지시라 코드실행 단계 없이 응답).

### 1. 인증 시스템 (자체 구현, 라이브러리 무의존)

Node.js 내장 `crypto`만으로 세션 기반 인증을 구현. 서버 세션 + HMAC 서명 쿠키 방식(JWT 아님 → 로그아웃/권한변경 즉시 반영).

| 파일 | 역할 | 핵심 |
|---|---|---|
| `src/lib/password.ts` (신규) | 비밀번호 해싱/검증 | `scrypt`(N=16384,r=8,p=1). 포맷 `scrypt$N$r$p$salt$hash`. `DUMMY_HASH`로 사용자 존재 유추 방어, `timingSafeEqual` 비교 |
| `src/lib/session-crypto.ts` (신규) | 세션 쿠키 서명 | `HMAC-SHA256`, 시크릿 `AUTH_SECRET`. 쿠키명 `pms_session`, 포맷 `<sessionId>.<hmac>`. DB/next-headers 무의존(proxy에서 재사용) |
| `src/server/session.ts` (신규) | 세션 생명주기 | `createSession`(TTL 7일, httpOnly·sameSite=lax·prod에서 secure), `getSessionUser`(쿠키검증→DB조회→만료검사), `destroySession` |
| `src/server/auth.ts` (수정) | 인증 진입점/가드 | `getCurrentUser`, `getScope`(현재 `where:{}` — 향후 테넌트 격리 seam), `requireUser`(→`/login`), `requireAdmin`(비관리자→`/`) |
| `src/proxy.ts` (신규) | Next 16 미들웨어 | 요청 쿠키의 HMAC만 검증(DB 접근X). 무효 시 `/login` 리다이렉트. matcher에서 `/login`·`/api`·`_next/**` 제외 |
| `src/app/login/page.tsx`, `src/components/auth/LoginForm.tsx` (신규) | 로그인 화면 | `login` 액션 호출 |
